<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Human/agent terminal -> real GitHub remote (documented, not executed, by this plan) | `RELEASE.md` documents commands that, when actually run (deferred to 05-04-PLAN.md), publish commits/tags/releases publicly. |
| `gh` CLI authenticated session -> GitHub Releases API | The runbook relies on the operator's own already-authenticated `gh auth login` session — no new credential is introduced by this plan. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Information Disclosure | `RELEASE.md`'s documented `git push origin main` step | accept | Publishing 79+ local commits (full internal migration + GSD-bookkeeping history) to the already-public `tpougy/finance-fmt-tools` repo is an accepted, expected outcome of shipping this milestone. `RELEASE.md` flags this explicitly near the top so whoever runs it later is not surprised; no secrets/credentials exist in that history (source + planning docs only). |
| T-05-05 | Repudiation | Manual `gh release create` run by a human or an agent, no additional audit trail beyond git/GitHub's own | accept | `gh`/git already provide an immutable, attributable audit log (commit author, `gh auth status` identity, GitHub's own release/Actions history) — sufficient for a solo-developer project; no additional logging needed. |
| T-05-06 | Tampering | `RELEASE_NOTES.md` content could drift from what a past release actually published if edited after the fact | accept | Documented, by-design behavior (05-RESEARCH.md Architecture Patterns): "once a release is created, its body text is permanent regardless of later file edits." Each new release simply overwrites `RELEASE_NOTES.md` with that version's own entry before tagging. |

</threat_model>
