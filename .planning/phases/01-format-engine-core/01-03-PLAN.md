**Pitfall — Spread bps quote-escaping:** VBA source is `f.NumberFmt = "#,##0.0"" bps"""`. VBA escapes a literal `"` by doubling it (`""`). Decoded character-by-character: `#,##0.0` + (`""`→literal `"`) + ` bps` + (`""`→literal `"`, closing the string) = the real runtime string `#,##0.0" bps"`. In a C# string literal this is `"#,##0.0\" bps\""`.

**Pitfall — Date BR Longa tooltip mismatch:** The Ribbon tooltip (`src/customUI14.xml:109-115`) shows `Ex.: 15 de março de 2025` (spelled-out month), but the actual VBA `NumberFmt` (`modFormatEngine.bas:150-154`) is `"[$-pt-BR]dd/mmm/yyyy;@"` — abbreviated 3-letter month, rendering `15/mar/2025`. Port the code's literal string, not the tooltip's description — this phase's goal is byte-for-byte parity with the shipped VBA, not "fixing" a tooltip/code drift.

**Critical finding — Integer is NOT `#,##0`:** `FMT_INTEGER`'s `NumberFmt` in VBA is literally `AccountingFmt(0, applyZeroDash:=CFG_ZERO_DASH)` — the same 3-section accounting logic as Fin2D/4D/8D, just `decimals=0`. There is no separate plain-integer code path. This is why `Integer` belongs in `FormatRegistryFinFamilyTests.cs`, not `FormatRegistryLiteralTests.cs`.

**Critical finding — Alignment is never set to Right anywhere in VBA:** Grep-confirmed across the entire `GetFormatDef` function (`src/modFormatEngine.bas:81-170`) — no `Case` branch ever assigns `f.Alignment`. The VBA `FormatDef` Type's `Alignment` field therefore always holds its uninitialized zero-value default (not `xlHAlignGeneral`, which is numerically `1`). The visual "right alignment" effect users see for Fin-family formats comes entirely from the `" * "` repeating-fill-character token that `AccountingFmt` already embeds into the `NumberFmt` string when `CFG_FORCE_ALIGN` is true (`modFormatEngine.bas:195-197`) — a number-format feature, not a `Range.HorizontalAlignment` COM property write. **Every `FormatDef` this registry constructs — including the Fin family — must use `CellAlignment.General`.** Do not set `CellAlignment.Right` for any entry (this would be a behavior change from the actual shipped VBA add-in, not a faithful port, even though 01-RESEARCH.md's illustrative Pattern 2 code sample shows `CellAlignment.Right` for the Fin family — that sample is illustrative pseudocode, not a literal spec, and contradicts the grep-verified VBA source, which takes precedence per this phase's byte-for-byte parity goal).
