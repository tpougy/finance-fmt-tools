## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| NuGet package supply chain → build output | Two required packages (`Microsoft.Office.Interop.Excel`, `MicrosoftOfficeCore16`) are third-party repackages of Office PIA binaries with no official Microsoft publisher badge and no stated license |
| Excel COM host → RealExcelGateway | `Application.Selection` is an untyped `object` returned by a foreign COM host — could be `Excel.Range`, `Excel.Chart`, `Excel.Shape`, or anything else the user has selected |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-03-01 | Tampering | `Microsoft.Office.Interop.Excel` / `MicrosoftOfficeCore16` NuGet packages | mitigate | Task 1's `checkpoint:human-verify gate="blocking-human"` requires explicit human sign-off (nuget.org registry check + `.nupkg` content inspection already performed by the researcher) before Task 2 adds either package — this is the required T-{phase}-SC gate for unofficial package sources, substituting `slopcheck` (unavailable, no `pip`) with direct registry/binary inspection per 03-RESEARCH.md's Package Legitimacy Audit |
| T-03-02 | Tampering | `RealExcelGateway.TryGetSelectedRange` | mitigate | Pattern-matched `sel is Excel.Range r` type check before any cast; non-`Range` COM objects (Chart/Shape/etc.) are never blindly treated as a Range and are explicitly released via `Marshal.ReleaseComObject` rather than left to ambiguous RCW finalization |
| T-03-03 | Information Disclosure | `TraceLog` | accept | Local, single-user desktop add-in; `System.Diagnostics.Trace` output only reaches listeners attached on the user's own machine (e.g. DebugView), and messages contain only cell addresses/format keys, no sensitive data |
</threat_model>
