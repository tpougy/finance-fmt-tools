<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Excel COM host → Connect | Office invokes `Connect`'s public methods by name via late-bound `IDispatch.GetIDsOfNames`, crossing from an untrusted, foreign process boundary into managed code |
| AddInHost.OpenDocs → OS shell | `Process.Start` opens a URL via the OS's default browser handler |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-03-04 | Denial of Service | `Connect` (all `IDTExtensibility2`/`IRibbonExtensibility`/`RibbonXxx` methods) | mitigate | Every method body wrapped in try/catch that logs via `TryLog` and never rethrows — an unhandled exception escaping a COM entry-point method can cause Excel's Resiliency system to silently disable the add-in (a Phase 4 concern too, `DoNotDisableAddinList`/INST-03), so this phase's own defensive coding is the first line of defense |
| T-03-05 | Tampering | `RibbonFinInfo` → `AddInHost.OpenDocs` → `Process.Start` | accept | URL is a hardcoded constant (`DocsUrl`), never derived from user input or external/untrusted data — no injection surface; `UseShellExecute=true` matches VBA's `FollowHyperlink` behavior exactly |
| T-03-06 | Spoofing | `Connect`'s `[Guid]`/`[ProgId]` registered under `HKCU\Software\Classes` | accept | GUID is a freshly-generated v4 UUID (`881EFDF3-424C-4240-BCA0-714DAC2B9CD7`); collision with another installed COM class is astronomically unlikely, and Phase 4 registers only under this add-in's own dedicated ProgID/CLSID keys |
</threat_model>
