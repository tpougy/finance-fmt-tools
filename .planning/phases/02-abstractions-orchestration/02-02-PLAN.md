**Embedded-resource loading via suffix match (Task 2) — from 02-RESEARCH.md Pattern 3, empirically verified in the research session (a scratch net8.0 console project embedding a file one directory above the project via Link, then enumerating Assembly.GetManifestResourceNames()):**
```csharp
public string GetCustomUiXml()
{
    Assembly asm = typeof(RibbonController).Assembly;
    string resourceName = null;
    foreach (string name in asm.GetManifestResourceNames())
    {
        if (name.EndsWith("customUI14.xml", StringComparison.OrdinalIgnoreCase))
        {
            resourceName = name;
            break;
        }
    }
    if (resourceName == null) return string.Empty;

    using (Stream stream = asm.GetManifestResourceStream(resourceName))
    using (StreamReader reader = new StreamReader(stream))
    {
        return reader.ReadToEnd();
    }
}
```
**Why suffix match, not a hardcoded full name:** SDK-style `EmbeddedResource` logical names are computed from `RootNamespace` + relative path, which silently drifts if either changes. A sibling project (`outlook-classic-delay-send`, this project's own dev/build/release inspiration per CLAUDE.md) shipped exactly this bug and had to add a regression test — resolve by `EndsWith(...)`, never by exact expected name (Pitfall 3).

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Embedded resource name resolution (`RootNamespace` + folder + `Link` path -> logical name) | Computed by the SDK at build time; drifts silently if project layout changes, with no build-time error — only a runtime empty-string symptom |
| `RibbonSessionConfig` mutation | In-memory only; any future caller mutating `Config.ForceAlign`/`ZeroDash` directly changes session behavior for every subsequent `FormatEngine` call in that session — by design, since there is no persistence to isolate from |
