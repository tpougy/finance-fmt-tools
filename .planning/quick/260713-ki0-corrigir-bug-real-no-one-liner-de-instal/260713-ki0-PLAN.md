<objective>
Corrigir o bug real (reproduzido ao vivo contra o GitHub raw hospedado, via WSL2↔Windows Excel
interop, usando o comando exato reportado pelo usuário) em que o one-liner documentado
`Set-ExecutionPolicy Bypass -Scope Process -Force; irm <url> | iex` falha com um erro de parse do
PowerShell ("Atributo 'CmdletBinding' inesperado" / "Token 'param' inesperado") porque
`Invoke-RestMethod` não remove o BOM UTF-8 inicial (`EF BB BF`) dos 3 arquivos em `scripts/`, e esse
BOM residual quebra o parser quando o conteúdo é passado para `Invoke-Expression`.

A correção decidida (não é para reinvestigar nem propor alternativa): tornar os 3 scripts PowerShell
inteiramente ASCII — sem nenhum byte com código > 127 — e remover o BOM. ASCII puro é um subconjunto
válido de toda code page e de UTF-8, então não sobra nenhuma ambiguidade de encoding para o PowerShell
interpretar mal; nenhum BOM é necessário, e o one-liner documentado ORIGINAL (sem nenhuma modificação,
sem `TrimStart`/tratamento de BOM) volta a funcionar tanto no fluxo `irm | iex` quanto no fluxo local
`-File`.

Purpose: eliminar de vez a causa raiz (ambiguidade de encoding), não mascarar o sintoma — nenhuma
mudança é feita em README.md, RELEASE.md, ou no texto do one-liner em si.

Output: `scripts/install.ps1`, `scripts/uninstall.ps1` e `scripts/verify-environment.ps1` sem BOM,
100% ASCII, com a mesma lógica/estrutura/contagem de linhas de hoje — apenas a prosa humana (comentários,
comment-based help, mensagens `Write-*`) transliterada de português-acentuado para ASCII.
</objective>
